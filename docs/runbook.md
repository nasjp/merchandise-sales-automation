# Runbook

この Runbook は `apps/web` の運用担当者が、日次運用と障害対応を同じ品質で実施するための手順書です。
対象は `raw_events` / `candidates` / `price_snapshots` / `task_audit` の運用です。

## 1. 監視指標と基準値

毎日最初に次の 4 指標を確認します。

| 指標 | 参照画面 | 正常の目安 | 異常判定 |
| --- | --- | --- | --- |
| Task 失敗率 | `/runs` | 5分平均 5% 未満 | 5% 以上が 10 分継続 |
| 候補生成数 | `/candidates` | 前週同曜日比 70% 以上 | 50% 未満 |
| Snapshot 更新 | `/targets` | 全ターゲットで24h以内に観測あり | 24h超の未更新が 1 件以上 |
| 受信遅延 | `raw_events` と `candidates` | p95 120 秒未満 | p95 120 秒超 |

しきい値超過時は、`5. 障害別手順` の該当項目に即時移行します。

## 2. 日次運用チェック

### 2-1. 始業時チェック（10分以内）

1. `/runs` で `failed` の件数と連続発生を確認する。
2. `/targets` で Price Snapshot が古いターゲットがないか確認する。
3. `/candidates` で未処理候補（未承認・未却下）の滞留件数を確認する。
4. `/settings` で Data Health の最新時刻が更新されているか確認する。

### 2-2. 日中運用

1. `/candidates` は未処理のみを基準としてレビューする。
2. 却下時は理由を必ず入力し、再学習に使える文言にする（例: 相場より高値、状態不明、型番不一致）。
3. `reason` が同一パターンで増えている場合は、閾値または抽出ロジックの改善チケットを起票する。

### 2-3. 終業前チェック

1. 未処理候補が SLA（24h）を超えていないか確認する。
2. 当日の障害対応があれば、原因・対処・再発防止をチケットに記録する。

## 3. 再実行運用（`/runs`）

再実行は「一時的失敗」のみ実施します。恒久障害のまま再実行しないでください。

### 実施条件

- 外部API一時エラー、タイムアウト、短時間の接続障害。
- 直近で設定変更を反映済みで、同 run の再評価が妥当。

### 手順

1. `/runs` で対象 run の `task` と `status` を確認する。
2. `再実行` を押し、`queued` になったことを確認する。
3. `running -> success` まで遷移を確認する。
4. 期待する更新先（`/candidates` または `/targets`）で結果を確認する。

### 実施しない条件

- 入力データ欠損、コード不整合、認証失敗など再発確実な原因。
- 同 run に対して 3 回連続失敗した場合（即エスカレーション）。

## 4. 障害レベル定義

| レベル | 条件 | 目標初動 |
| --- | --- | --- |
| Sev1 | 候補生成停止、または ingest 停止 | 5 分以内 |
| Sev2 | 失敗率上昇、Snapshot 停滞、レビュー滞留急増 | 15 分以内 |
| Sev3 | 部分劣化、単発失敗、軽微な遅延 | 当日中 |

## 5. 障害別手順

### 5-1. ingest 401 / 認証失敗増加

1. `ANDROID_INGEST_SHARED_SECRET` の差分反映漏れ有無を確認する。
2. Android 側送信時刻と web 側受信時刻のズレを確認する。
3. ローテーション作業中なら旧新キー切替順序を再確認する。
4. 復旧後、15 分間 401 が再発しないことを確認する。

### 5-2. task failed 増加

1. 失敗 run の `taskName` と `payload` を分類する（同一原因かを判定）。
2. 外部依存（DB / Trigger API / LLM）起因か、アプリ起因かを切り分ける。
3. 一時障害なら `3. 再実行運用` を実施する。
4. 恒久障害なら修正PRを先に作成し、デプロイ後に再実行する。

### 5-3. Snapshot 停止

1. `/targets` で `Observed At` が古いターゲットを抽出する。
2. 対象ターゲット単位で `相場更新` を実行し、更新可否を確認する。
3. 複数ターゲットで同時失敗ならジョブ基盤障害として Sev2 扱いにする。

### 5-4. 候補生成数の急減

1. `raw_events` 件数が同時に減っているか確認する。
2. `raw_events` が正常で `candidates` のみ減少なら抽出・マッチング側を確認する。
3. `review_state = excluded` の比率急増を確認し、閾値劣化を疑う。

## 6. 復旧完了の定義

次を満たした時点で「復旧」とします。

1. 監視指標 4 項目が 30 分連続で基準内。
2. 失敗 run の増加が停止。
3. 未処理候補の増加傾向が収束。

## 7. エスカレーション

以下は即時エスカレーション対象です。

1. Sev1 事象。
2. 同一原因で 30 分以内に再発。
3. 再実行 3 回で復旧しない失敗。

エスカレーション時は、最低限次を共有します。

- 発生時刻（JST）
- 影響範囲（候補生成 / Snapshot / ingest）
- 現在の暫定対処
- 恒久対処の着手状況（Issue/PR）
